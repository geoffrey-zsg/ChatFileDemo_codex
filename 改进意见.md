# 改进意见

## 2026-03-12

### 项目初始化阶段
- `template_docs` 提到了 `docs/PLANS.md`，但模板列表未直接给出，实际初始化时需要补建，建议模板仓库显式提供。
- 参考文档中的交互要求很细，但实现优先级未二次分层；后续可补一个“P0 UI 细化清单”，减少 AI 在取舍时的主观偏差。
- 经验教训里强调截图验收，这条应直接前置进模板化的 `QUALITY.md`，避免后续遗漏。
- `template_docs/root` 目前缺一个更偏“项目启动态”的模板，例如 `PLANS.template.md` 或 `BOOTSTRAP.template.md`，用于明确当前阶段目标、默认假设、短期里程碑和阻塞处理方式。
- `template_docs/docs/quality.template.md` 建议增加“截图验收要求”“端口/服务探测要求”“失败后如何重跑与定位”的固定章节，让 AI 不会只跑命令不看页面。
- `template_docs/docs/frontend.template.md` 建议补一节“Mock 边界定义”，说明哪些能力允许先 mock、哪些交互即便 mock 也必须完整，例如上传状态、错误态、流式输出、引用联动。
- `template_docs/docs/architecture.template.md` 建议补一个“空仓库初始化分流”章节，区分“已有工程迭代”和“从零搭骨架”两种模式，避免 AI 一上来按已有项目方式查找不存在的代码。
- `template_docs/root/AGENTS.template.md` 里的多 Agent 协作规则可以更具体些，建议补“何时不该并行”“共享契约先行顺序”“端口和开发服务归谁负责”的说明，减少多 Agent 重复启动服务。
- `template_docs` 可以增加一个轻量的 `ACCEPTANCE.template.md`，把验收项、测试命令、截图路径、已知风险汇总到一页，方便 AI 每轮收尾时复用。

### 协作机制
- 当前仓库几乎是空目录，AI 首次进入时应先判断“是规范初始化任务还是功能迭代任务”，这个分流规则值得沉淀。
- 子 Agent 并行适合放在共享契约已稳定之后；在架构未定前并行容易返工，建议后续形成更明确的触发条件清单。
- e2e 执行前应先探测端口占用与服务健康，而不只是判断“端口是否被监听”；本次 3100 端口上存在挂起但不可用的 dev 进程，导致浏览器测试误判。
- Playwright 若依赖外部已启动服务，建议将“自动起服务”和“复用现有服务”拆成两种明确模式，减少端口冲突和启动超时。

### 第一轮自主推进复盘
- 第一轮我采用的是“先串行定契约，再并行收集上下文，最后再集中实现”的推进方式，没有一开始就硬拆多个子 Agent。
- 实际分工上，我把工作拆成四段：`参考文档分析`、`模板文档初始化`、`前端工程与 mock BFF 搭建`、`自动化校验和截图验收`。这四段之间依赖明显，所以主干采用串行推进。
- 并行部分主要用在信息收集和校验阶段，而不是代码实现阶段：例如并行读取参考文档、并行检查模板文件、并行运行 `lint/typecheck/test`、并行探测端口和进程。这相当于“轻量子 Agent 化”，但避免了多 Agent 同时写共享文件。
- 第一轮我没有再额外创建专门负责 UI、数据、测试的独立子 Agent，原因是仓库起点接近空白，架构、目录和契约都还没稳定；如果这时并行写代码，返工概率会很高。
- 真正用了 skills 的地方是 `frontend-design`，它在第一轮主要承担“界面风格约束”和“避免默认模板味”的角色，而不是独立实现代码。
- 自主推进顺序是：先读 `docs/refer` 明确目标和教训，再初始化规范文档，把质量门槛和验收方式写死，然后直接搭工程和主链路，最后用自动化测试加截图兜底。这种顺序能减少“先写页面、后补规范”导致的返工。
- 第一轮暴露出的不足是：浏览器测试服务探测做得还不够早，导致在端口冲突和挂起服务上多花了一轮定位时间。这个问题已经补充进本文件，也会在第二轮继续按“先探测服务健康再跑测试”的方式执行。

### 后续自主推进复盘
- 第二轮到最终收尾阶段，我采用的是“先补用户侧可见价值，再补产品感，再做最终稳态验证”的顺序，优先补了拖拽上传、反馈面板、多轮追问、无结果态，再补帮助层、快捷键、预览页码跳转和缩放。
- 这一阶段依然没有把代码实现拆成多个真正独立的子 Agent，原因是主界面仍然高度集中在 `src/app/page.tsx` 和几个核心组件里，强行并行会增加冲突和集成成本。
- 真正适合并行的部分仍然是验证和诊断：我把 `typecheck/lint/test/build`、端口探测、服务健康检查、错误上下文读取这些动作并行处理，用来缩短定位时间。
- 后半段最典型的问题不是功能逻辑，而是环境一致性：旧的 dev server 继续占用 3300 端口，而 `.next` 目录已被新的构建过程重写，导致浏览器访问时报 vendor chunk 丢失。这说明“端口可访问”不等于“服务健康”。
- 针对这个问题，我最终采用了更稳的处理顺序：先探测端口和 HTTP 健康，再读取错误上下文，再清理 `.next`，最后重启干净服务并重新跑 e2e。这个顺序比一味重试更高效，也更适合作为模板化流程。
- 后续如果要把这套机制沉淀得更强，建议在模板文档中增加“开发服务生命周期”约定：谁负责启动服务、何时复用、何时必须重启、何时应清理构建产物。
- 从自主推进效果看，最有价值的不是拆出更多 Agent，而是尽早固定验收闭环和诊断路径。规则一旦清楚，AI 连续推进的稳定性会明显高于临时问答式协作。

### UI 幻觉复盘
- 这次最大的问题确实是我对页面 UI 质量产生了错误判断。原因不是单一的“审美不够”，而是我在样式系统未真正生效的情况下，过早把注意力放到了功能完整性上。
- 具体根因有两个：第一，Tailwind 管线最初并没有正确产出 utility class；第二，我虽然做了浏览器截图，但没有把截图当成“视觉验收”来审，而是更偏向确认功能链路是否跑通，导致明显的裸页面信号没有被及时升级处理。
- 这暴露出一个流程问题：功能验收截图和视觉验收截图不能混为一类。以后应明确分成两步，先看“功能是否通”，再单独看“视觉是否达到产品级”。
- 后续修正路径是：先定位样式基础设施问题，再恢复 Tailwind 内容扫描，再重启干净服务重新截图，最后才做颜色、层次和排版的视觉 polish。这个顺序比直接改 JSX 样式更有效。
- 如果把这个教训写进模板，建议在 `QUALITY` 或单独的验收模板中加入一句硬性要求：`截图必须由 AI 进行视觉复核，若页面仍接近裸 HTML、无明显层级或布局失衡，不得判定通过。`
