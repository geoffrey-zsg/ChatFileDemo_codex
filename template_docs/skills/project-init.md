# Skill: project-init

> 基于模板和技术栈信息，初始化项目专属的 AI 协作文档体系

## 适用场景

- 新项目启动，需要建立 AI 协作文档
- 已有项目，首次引入 AI 协作文档体系
- 技术栈已确定，需要将通用模板具体化

## 不适用场景

- 文档已经初始化完成，只是小修改
- 技术栈尚未确定
- 只是想了解模板内容

## 输入

用户需要提供：
- **技术栈信息**：框架、语言、构建工具、包管理器
- **核心技术选型**：状态管理、样式方案、路由、表单、测试
- **项目背景**：项目类型、团队规模、项目周期
- **计划使用的第三方 Skills**：列出所有计划使用的 skills

需要读取的模板文档：
- `templates/root/AGENTS.template.md`
- `templates/root/CLAUDE.template.md`
- `templates/docs/frontend.template.md`
- 其他根据配置选择的模板

## 执行步骤

1. **收集信息**
   - 询问用户技术栈和项目背景
   - 确认已复制的模板文档列表
   - 获取计划使用的第三方 Skills 列表

2. **读取模板**
   - 读取用户已复制的所有模板文档
   - 理解模板结构和占位符

3. **填充文档**
   - 将通用占位符替换为具体技术栈
   - 补充技术栈相关的约束和规范
   - 添加常用命令和工具链配置

4. **冲突检测**
   - 分析第三方 Skills 的核心建议
   - 识别与项目技术选型的冲突点
   - 列出所有冲突项

5. **决策引导**
   - 对每个冲突点提出清晰的选项
   - 说明每个选项的优缺点
   - 等待用户决策

6. **应用决策**
   - 根据用户决策更新文档
   - 在文档中明确记录决策和理由
   - 对不采用的第三方建议，添加说明

7. **一致性检查**
   - 检查所有文档的技术栈描述是否一致
   - 检查文档之间是否有内部冲突
   - 生成一致性检查报告

8. **生成总结**
   - 列出已填充的文档
   - 列出已解决的冲突
   - 提供后续维护建议

## 输出

- 已填充的项目文档（AGENTS.md、CLAUDE.md、FRONTEND.md 等）
- 冲突解决记录
- 一致性检查报告
- 后续维护建议

## 注意事项

- 不要自作主张解决冲突，必须让用户决策
- 填充时保持文档简洁，避免过度详细
- 技术栈相关的约束要具体、可执行
- 记录决策理由，方便后续回溯
- 提醒用户配置反馈闭环（lint、test、CI）
