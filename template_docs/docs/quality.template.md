# QUALITY.md

> 定义项目质量门槛。让人和 AI 都知道什么叫"可以合并"。

## 合并前最低要求

```bash
# 必须通过
pnpm lint          # 代码规范检查
pnpm typecheck     # 类型检查
pnpm test          # 单元测试

# 特定场景必须通过
pnpm test:e2e      # 核心流程改动时
```

## 通用质量标准

### 性能（可根据项目调整）
- 首屏 LCP < 2.5s
- 交互响应 < 300ms
- Bundle Size 增量 < 50KB（需说明原因）

### 可访问性
- 表单控件必须有 label
- 键盘可操作（Tab、Enter、Esc）
- 颜色对比度 >= 4.5:1
- 错误提示可理解（不只说"失败"）

### 代码质量
- 无 TypeScript `any`（除非有充分理由）
- 无 ESLint error（warning 需评估）
- 公共函数有注释说明
- 复杂逻辑有单元测试

### 用户体验
- 所有异步操作有 loading 状态
- 所有请求有 error 处理
- 空状态有提示和引导
- 危险操作有二次确认

## 测试策略

### 单元测试
- 纯函数、工具函数
- 复杂 hooks
- 业务逻辑计算

### E2E 测试
- 核心用户路径（登录、下单、支付）
- 关键业务流程
- 跨页面交互

## 可观测性

- 关键错误上报（Sentry/监控平台）
- 核心流程埋点（成功/失败）
- 性能指标监控（Web Vitals）

---

## 示例：管理后台项目

```markdown
## 合并门槛
- 所有改动：`pnpm lint && pnpm typecheck && pnpm test`
- 订单/退款/权限相关：额外运行 `pnpm test:e2e`

## 性能要求
- 首屏可交互时间 < 2.5s
- 列表筛选响应 < 300ms
- 大列表使用虚拟滚动（>1000 条）

## 可观测性
- 接口错误统一通过 `errorHandler` 上报
- 关键操作埋点：订单创建、退款审核、批量导出
```
