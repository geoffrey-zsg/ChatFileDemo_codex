# DESIGN.md

## 1. 文档目的
统一 ChatFile Demo 的视觉语言、组件反馈和响应式策略，减少 AI 在 UI 实现上的随意性。

## 2. 设计原则
- 信息密度高但层级清晰。
- 企业工具感优先于装饰性。
- 对话区与预览区视觉区分明确。
- 所有关键动作和异常都要有即时反馈。
- 保持需求文档给出的布局与交互主干。

## 3. 设计令牌
- 颜色：基于暖白底 + 石墨文本 + 青蓝强调色 + 柔和高亮色。
- 字号：标题 18/16，正文 14，辅助 12，代码 13。
- 间距：以 4px 基础网格递增。
- 圆角：8 / 12 / 24 三档。
- 阴影：输入区和浮层使用轻阴影，对话气泡不依赖厚阴影。
- 动效：默认 150ms 到 300ms，引用高亮和流式输出允许更显著。

## 4. 组件规范
- 按钮：区分主、次、幽灵、危险、图标按钮；加载中禁止重复点击。
- 输入框：必须有禁用态、聚焦态、超长态。
- 抽屉/面板：目录、反馈、帮助都采用统一浮层规则。
- 空状态：给出下一步动作，不只是一句提示。
- Toast：只做瞬时反馈，不替代表单内错误。

## 5. 页面布局规范
- 默认左右分栏 4:6，允许拖拽和预览折叠。
- 顶部导航固定展示应用名和当前文档信息。
- 底部输入区固定悬浮，保证对话区可持续操作。

## 6. 状态设计
- 上传中：进度条和取消入口同时可见。
- 解析中：步骤条明确当前阶段和预计耗时。
- 就绪：展示横幅、摘要和推荐问题。
- 错误：提供可执行恢复动作。
- 生成中：输入框禁用，按钮切为停止态。

## 7. 响应式与适配
- `>= 1280px`：双栏完整模式。
- `768px - 1279px`：预览区可切换抽屉化。
- `< 768px`：默认单栏，预览作为覆盖层打开。
- 长文档名、长引用、长问题文本都要做截断和展开策略。

## 8. 文案与反馈
- 文案短、直接、动作明确。
- 无法回答时要明确是“当前文档中未找到”。
- 危险动作和清空对话需要二次确认。
